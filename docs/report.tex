\section{Introduction}
%% ════════════════════════════════════════════════════════════════════════════

The \textbf{0/1 Knapsack Problem} is a canonical NP-Hard combinatorial optimisation
problem. Given $n$ items, each with weight $w_i$ and value $v_i$, and a knapsack of
capacity $W$, we must choose a subset of items that maximises total value while
respecting the capacity constraint. The ``0/1'' qualifier forbids fractional
selections: each item is either taken whole or left behind. Formally:
\begin{equation}
\max \;\sum_{i=1}^{n} v_i x_i \quad
\text{subject to}\quad \sum_{i=1}^{n} w_i x_i \le W,\quad x_i \in \{0,1\}.
\label{eq:knap}
\end{equation}

\textbf{Real-world applications.} Capital budgeting, cargo loading, cloud VM
placement, ad-slot allocation, portfolio selection, and project prioritisation
all reduce to (\ref{eq:knap}). The problem is \emph{weakly} NP-Hard: a
pseudo-polynomial $O(nW)$ dynamic-programming solution exists, but $W$ requires
$O(\log W)$ bits, so the algorithm is exponential in the input bit length.

\textbf{What this project does.} We implement, benchmark and compare \textbf{six}
algorithms for 0/1 Knapsack: \emph{Brute Force}, \emph{Memoization (top-down DP)},
\emph{Tabulation (bottom-up DP)}, \emph{Space-Optimised DP}, a \emph{Greedy
$\tfrac{1}{2}$-approximation}, and a \emph{Fully Polynomial-Time Approximation
Scheme (FPTAS)}. All algorithms are pure Python 3 with no third-party
dependencies, so observed differences reflect algorithmic structure rather than
library tuning.

\subsection{Brute Force (Exhaustive Recursion)}
For each item, recursively branch into ``include'' and ``exclude''. All $2^n$
subsets are implicitly enumerated and the best feasible one returned.
\begin{equation}
\mathrm{BF}(i,w)=\begin{cases}
0 & i=0\;\text{or}\;w=0\\
\mathrm{BF}(i-1,w) & w_i>w\\
\max\bigl(\mathrm{BF}(i-1,w),\;v_i+\mathrm{BF}(i-1,w-w_i)\bigr) & \text{otherwise}
\end{cases}
\end{equation}
\textit{Use:} correctness oracle for small $n$; explains why DP is needed
(identical subproblems are recomputed exponentially many times).

\subsection{Memoization (Top-Down DP)}
The same recurrence as Brute Force, but each subproblem $(i,w)$ is cached on first
computation. The state space has $(n+1)(W+1)$ cells, each filled in $O(1)$ work,
giving $O(nW)$ time. Sentinel \texttt{None} (not $-1$) is used so legitimate zero
values are not mistaken for cache misses. \textit{Use:} when the recursion only
visits a sparse subset of states; preserves the natural recursive structure.

\subsection{Tabulation (Bottom-Up DP)}
Build a 2-D table $\mathrm{dp}[i][w]$ iteratively from base cases. The recurrence
is the standard knapsack recurrence:
\begin{equation}
\mathrm{dp}[i][w]=\begin{cases}
0 & i=0\;\text{or}\;w=0\\
\mathrm{dp}[i-1][w] & w_i>w\\
\max\bigl(\mathrm{dp}[i-1][w],\,v_i+\mathrm{dp}[i-1][w-w_i]\bigr) & \text{otherwise}
\end{cases}
\label{eq:dp}
\end{equation}
The selected items are recovered by backtracking from $\mathrm{dp}[n][W]$:
whenever $\mathrm{dp}[i][w]\neq\mathrm{dp}[i-1][w]$, item $i$ is in the optimal
set and we move to $(i-1,\,w-w_i)$. \textit{Use:} default exact solver in
production; cache-friendly, no recursion-stack risk, supports item recovery.

\subsection{Space-Optimised DP (1-D Array)}
Row $i$ in (\ref{eq:dp}) depends only on row $i-1$, so a single 1-D array
$\mathrm{dp}[w]$ suffices. The inner loop \emph{must} run from $W$ down to $w_i$:
\begin{equation}
\mathrm{dp}[w]\gets \max\bigl(\mathrm{dp}[w],\;v_i+\mathrm{dp}[w-w_i]\bigr),
\qquad w=W,W-1,\ldots,w_i.
\end{equation}
A forward loop would let an item be chosen multiple times, silently solving the
\emph{unbounded} knapsack instead. \textit{Use:} when $W$ is large but the
selected item set is not needed.

\subsection{Greedy Approximation (variant $S_2$)}
Sort items by value-to-weight ratio $r_i=v_i/w_i$ in descending order, pack
greedily, then return $\max(\mathrm{greedyValue},\,v_{\max})$ where $v_{\max}$
is the most-valuable single item that fits alone. This single tweak --- due to
Dantzig --- guarantees the result is at least $\tfrac{1}{2}\,\mathrm{OPT}$.
Without it, a single high-value item can be missed and the ratio is unbounded.

\begin{algorithm}[H]
\caption{Greedy $S_2$ approximation}
\small
\begin{algorithmic}[1]
\State sort items by $v_i/w_i$ descending
\State $G\gets 0,\ \mathrm{cap}\gets W$
\For{each item $i$ in sorted order}
    \If{$w_i\le \mathrm{cap}$} $G\gets G+v_i$;\ $\mathrm{cap}\gets\mathrm{cap}-w_i$ \EndIf
\EndFor
\State \Return $\max(G,\,\max\{v_i:w_i\le W\})$
\end{algorithmic}
\end{algorithm}

\textit{Use:} when $n$ is huge or $W$ is astronomical and a fast, ``good enough''
answer is acceptable.

\subsection{FPTAS (Fully Polynomial-Time Approximation Scheme)}
Given an error parameter $\varepsilon\in(0,1)$, the FPTAS scales values down by
$K=\varepsilon\,v_{\max}/n$, then runs a profit-indexed DP whose state is the
\emph{value} achieved (not the weight). Letting
$\tilde v_i=\lfloor v_i/K\rfloor$ and $V=\sum_i \tilde v_i$:
\begin{equation}
\mathrm{minW}[v]=\min_{S\subseteq[n],\,\sum_{i\in S}\tilde v_i=v}\sum_{i\in S}w_i.
\end{equation}
The answer is the largest $v$ with $\mathrm{minW}[v]\le W$, scaled back. Running
time is $O(n^2 V)=O(n^3/\varepsilon)$, and the returned value is at least
$(1-\varepsilon)\mathrm{OPT}$. \textit{Use:} when both exactness guarantees and
polynomial-time scaling in $n$ and $1/\varepsilon$ are required --- particularly
attractive when $W$ is large but values are not.

\section{Theoretical Complexity Analysis}
%% ════════════════════════════════════════════════════════════════════════════

\begin{table}[H]
\centering
\caption{Best, average, and worst-case complexity per algorithm. ``Worst input''
identifies the instance structure that triggers the worst case.}
\small
\begin{tabular}{@{}lcccp{5.0cm}@{}}
\toprule
\textbf{Algorithm} & \textbf{Best} & \textbf{Average} & \textbf{Worst} & \textbf{What triggers worst case} \\
\midrule
Brute Force        & $O(2^n)$ & $O(2^n)$ & $O(2^n)$ & every input ($n$ alone) \\
Memoization        & $O(n+W)$ & $O(nW)$ & $O(nW)$ & dense reachable state space \\
Tabulation         & $O(nW)$  & $O(nW)$ & $O(nW)$ & always fills the table \\
Space-Opt.\ DP     & $O(nW)$  & $O(nW)$ & $O(nW)$ & always fills the array $n$ times \\
Greedy $S_2$       & $O(n\log n)$ & $O(n\log n)$ & $O(n\log n)$ & runtime input-independent \\
FPTAS              & $O(n^2/\varepsilon)$ & $O(n^3/\varepsilon)$ & $O(n^3/\varepsilon)$ & large value range, small $\varepsilon$ \\
\bottomrule
\end{tabular}
\end{table}

\textbf{Why the DP recurrence works.} Knapsack exhibits the two ingredients DP
requires. (i) \emph{Optimal substructure:} if the optimum on items $1{\ldots}n$
includes item $n$, the rest is the optimum on items $1{\ldots}n{-}1$ with capacity
$W{-}w_n$ (cut-and-paste argument). (ii) \emph{Overlapping subproblems:} the
recursion tree visits the same $(i,w)$ pair from many paths, so caching collapses
$2^n$ recomputations into at most $(n{+}1)(W{+}1)$ unique states.

\textbf{Why Brute Force is hardest on dense, deep recursion.} Its runtime is
controlled solely by $n$ --- input values and weights only marginally affect
pruning. Hence its best, average, and worst case all coincide at $\Theta(2^n)$.

\textbf{Why Greedy $S_2$ is best on uncorrelated, worst on correlated.} When
$v_i$ and $w_i$ are independent, ratio ordering closely matches the LP relaxation
and the fractional gap is small. When values and weights are tightly correlated
(strongly correlated, spanner, almost-strongly correlated), every item has a
similar ratio so the ratio order carries little signal --- packing decisions
become almost arbitrary and the gap widens (see \S\ref{sec:results}).

\textbf{Why FPTAS trades $n$ for $W$.} Standard DP is exponential in $\log W$.
The FPTAS instead spends time polynomial in $n$ and $1/\varepsilon$ but
\emph{independent of $W$} --- crucial when the capacity dominates the input size.

\subsection{Test-Case Strategy}
We use the Pisinger \emph{kplib} benchmark suite~\cite{pisinger2003,kplib}, which
is the de-facto standard for knapsack experimentation. Categories cover
\textbf{Uncorrelated} ($w_i,v_i$ independent), \textbf{Weakly} and
\textbf{Strongly Correlated} ($v_i \approx w_i + c$), \textbf{Inverse} and
\textbf{Almost-Strongly Correlated}, \textbf{Subset Sum} ($v_i = w_i$),
\textbf{Spanner} variants, \textbf{Multiple Strongly Correlated},
\textbf{Profit Ceiling}, \textbf{Circle}, plus a \textbf{13Synthetic} family we
generated ourselves for $n\in\{5,8,10,12,15,18,20,22,25\}$ to keep brute force
feasible. We benchmark over two coefficient ranges: \textbf{R01000} ($R{=}1000$)
and \textbf{R10000} ($R{=}10000$); capacity is set to
$W=\lfloor\tfrac{1}{2}\sum_i w_i\rfloor$, the standard Pisinger setting.

\subsection{Benchmarking Methodology}
\begin{itemize}[noitemsep,leftmargin=*]
\item \textbf{Single-process, fresh interpreter} per benchmark file --- no warm-up
      effects between algorithms.
\item Each algorithm is timed individually on each instance with
      \texttt{time.perf\_counter()}; values returned are checked for agreement
      across the four exact algorithms (correctness oracle).
\item \textbf{Skip thresholds} (\texttt{benchmark.py} only) prevent OOM/runaway
      runs: BruteForce $n>25$, Memoization $nW>15\!\times\!10^6$, Tabulation
      $nW>30\!\times\!10^6$, SpaceOpt $W>2\!\times\!10^6$, FPTAS $n^2/\varepsilon>
      4\!\times\!10^6$, Greedy never. \texttt{benchmark\_n100.py} and
      \texttt{benchmark\_r01000.py} remove all skips except Brute Force, ensuring
      DP variants are measured on every reachable instance.
\item Averaging: $\ge 3$ instances per (category, $n$, ratio) leaf; up to 100 in
      the kplib originals. Reported numbers below are arithmetic means.
\item \textbf{Approximation quality} for Greedy and FPTAS is computed as the
      ratio (returned value)$/$(exact DP value) per instance and aggregated by
      category.
\item Raw output: \texttt{results/benchmark\_n100.csv} (7\,435 rows) and
      \texttt{results/benchmark\_r01000.csv} (900 rows) are checked into the
      repository for reproducibility.
\end{itemize}

\subsection{Correctness}
On the standard baseline ($n{=}4$, $v{=}[60,100,120,80]$, $w{=}[10,20,30,25]$,
$W{=}50$, OPT$=220$), the four exact algorithms all return 220 and select items
$\{B,C\}$ ($w{=}50$, $v{=}220$). Greedy returns 160 (a 27.3\% gap): it commits to
items A and B by ratio, then cannot fit C. FPTAS at $\varepsilon{=}0.1$ returns
220. Across the entire kplib sweep, the four exact algorithms agreed on every
single instance.

\subsection{Edge Cases and Worst-Case Inputs}
\begin{itemize}[noitemsep,leftmargin=*]
\item \textbf{Brute Force:} every $n>25$ is a worst case; runtime is
      input-shape-insensitive. We observed $\sim\!140$ ms at $n{=}20$ vs.\
      $\sim\!4.3$ s at $n{=}25$, a $31\times$ blow-up for a $5$-item increase ---
      the signature of $2^n$.
\item \textbf{All DP variants:} blow up when $W$ is huge irrespective of $n$.
      At R10000 (10$\times$ larger weights) DP runtimes grow $\sim 5$--$10\times$,
      consistent with $O(nW)$.
\item \textbf{Greedy:} worst on \emph{spanner} and \emph{strongly correlated}
      instances, where ratio ordering carries little information; min ratio
      $0.586$ on SpannerWeakly. Best on uncorrelated and inverse-strongly
      correlated, where ratios are well-separated.
\item \textbf{FPTAS:} worst on instances with very few items but large value
      spread (high $v_{\max}/n$), where the scaling factor $K$ loses precision;
      this is reflected in 13Synthetic (small $n$).
\end{itemize}

\subsection{Capacity-Ratio Impact}
Doubling the value range (R01000 $\to$ R10000) leaves Brute Force, Greedy and
FPTAS runtimes essentially unchanged --- as predicted, none of them depend on
$W$. DP runtimes grow proportionally to $W$, confirming the
\emph{pseudo-polynomial} nature of $O(nW)$.

\subsection{Strengths and Weaknesses Summary}
\begin{table}[H]
\centering
\small
\begin{tabular}{@{}p{2.6cm}p{5.4cm}p{5.4cm}@{}}
\toprule
\textbf{Algorithm} & \textbf{Strengths} & \textbf{Weaknesses} \\
\midrule
Brute Force        & always exact; trivial to implement; correctness oracle for tests
                   & exponential; infeasible $n>25$ \\
Memoization        & top-down; exploits sparse state spaces; natural recursive form
                   & Python recursion overhead; $O(nW)$ memory \\
Tabulation         & no recursion; cache-friendly; supports item backtracking
                   & $O(nW)$ memory; expensive when $W$ is large \\
Space-Opt.\ DP     & only $O(W)$ memory; same time as tabulation
                   & no item recovery; reverse-loop discipline required \\
Greedy $S_2$       & $O(n\log n)$; W-immune; $\ge\tfrac{1}{2}$OPT guaranteed
                   & arbitrarily bad on adversarial correlated instances \\
FPTAS              & user-tunable $(1{-}\varepsilon)$ guarantee; W-immune
                   & $1/\varepsilon$ blowup; slower than DP for moderate $n,W$ \\
\bottomrule
\end{tabular}
\end{table}

%% ════════════════════════════════════════════════════════════════════════════
\section{Conclusion}
%% ════════════════════════════════════════════════════════════════════════════

The 0/1 Knapsack Problem is a microcosm of algorithmic trade-offs. Across more
than $7{,}000$ benchmark runs spanning $13$ Pisinger instance families, two
coefficient ranges and item counts from $5$ to $200$, our six implementations
behaved exactly as theory predicts: Brute Force is $\Theta(2^n)$ regardless of
input shape; the three DP variants are $\Theta(nW)$ with Tabulation winning at
moderate $n$ thanks to cache behaviour and Space-Optimised DP winning on memory;
Greedy $S_2$ is essentially free but loses up to $41\%$ on adversarial spanner
instances; FPTAS gives a tunable $(1-\varepsilon)$ guarantee at the cost of
$1/\varepsilon$ overhead. The practical recommendation is to default to
Tabulation when item recovery is needed, switch to Space-Optimised DP when only
the value matters and $W$ is the binding constraint, and reach for FPTAS or
Greedy $S_2$ only when both $n$ and $W$ are too large for $O(nW)$ to fit in time
or memory budgets.
